## Researcher

**Q1 — Position : Option B (heartbeat léger + T-3 mois)**

L'ARCHITECTURE.md §10 affirme que l'absence de rapport mensuel est « visible » — le stress-test a établi le contraire. C'est une contradiction documentée. Les modes de mort silencieux du cron (keychain, auth expirée, CLI auto-update) sont réels, multiples, et le coût d'un run réussi est 60-120 $ USD/an pour vérifier des restaurants qui ferment saisonnièrement en hiver de toute façon. Le heartbeat Actions répond au seul besoin réel pendant la dormance : garder le repo actif. Le reste appartient au jalon T-3 mois.

**Q2 — Position : migrer les deux, mais séquencés (Crète d'abord, Turquie immédiatement après)**

Crète : 2711 lignes. Turquie : 2749 lignes. Delta de 38 lignes — même template, même structure. Les deux ont eu 2 passes de corrections post-création sur les mêmes classes de bogues. Laisser Turquie en v1 crée une surface de maintenance dédoublée pour 95 % de code identique. Et les IDs Unsplash hotlinkés dans Turquie vont reporir — le workflow de correction sera différent entre v1 et v2, ce qui est exactement le problème qu'on essaie d'éliminer.

L'argument YAGNI a une validité partielle : migration ≠ création neuve, et la vraie preuve de répétabilité est la création d'une nouvelle destination. Mais différer Turquie à T-3 mois 2027 = ajouter une migration de site live au stress d'un pré-voyage. Mauvais timing. Migrer pendant que le contexte est frais.

## Architect

**Q1 : B** (heartbeat léger + validation manuelle T-3 mois)

Le cron mensuel est une fiction de monitoring : tous ses modes de mort sont silencieux, et le fait établi « l'absence d'un rapport mensuel n'est jamais remarquée » confirme que le feedback loop est brisé. B sépare proprement deux problèmes distincts — garder repo/Pages vivants (heartbeat Actions, trivial, fiable) vs fraîcheur des données (session T-3 mois, en saison, quand le $5-10/run a une valeur réelle). Payer 12 runs/année pour du bruit hivernal + fermetures saisonnières n'est pas de la diligence.

**Q2 : B** (Crète seulement, Turquie reste v1 statique) — **avec condition**

La 2e migration ne prouve pas la répétabilité du skill v2 — elle prouve que la migration fonctionne. Le chemin critique est la création d'une destination neuve. La cohabitation v1/v2 dans le repo est gérable (pas de collision build, Astro ignore le HTML statique), mais la Turquie v1 reste hors pipeline de validation.

Condition non négociable : documenter explicitement dans ARCHITECTURE.md l'état par destination et la date cible de migration Turquie (T-3 mois 2027). Sans ça, la dette cognitive se matérialise comme une décision à re-prendre à chaque session dormante — c'est un bug de documentation qui coûte autant qu'un bug de code.

## Implementer

La raison correcte n'est pas « proof of repeatability » mais **« une seule interface de maintenance »** — à T-3 mois 2027, revalidate-businesses.sh s'attend à un pois.json, pas à du HTML v1. Estimation effort 2e migration : **3-4 heures** une fois les extracteurs écrits pour Crète.
